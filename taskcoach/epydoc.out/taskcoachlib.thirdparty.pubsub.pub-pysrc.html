  1  ''' 
  2  This is the main entry-point to pubsub's core functionality. The pub module  
  3  supports: 
  4   
  5  * messaging: publishing and receiving messages of a given topic 
  6  * tracking: tracking certain pubsub calls in an application 
  7  * trapping exceptions: dealing with "badly behaved" listeners (ie that leak exceptions) 
  8  * specifying topics: defining (or just documenting) the topic tree of an  
  9    application; pubsub will verify that your senders and listeners adhere to 
 10    to it 
 11   
 12  The ``pub`` module also exposes several 
 13  classes such as Topic, Listener, Publisher, and many more that could 
 14  be used in advanced pub-sub applications. 
 15   
 16  Note that many functions in this module actually make use of an  
 17  instance of Publisher and one of TopicManager, both created upon import. 
 18   
 19  About topic definition tree: 
 20   
 21  Using ``pub.sendMessage('your.topic.name', **kwargs)`` is nice but in bigger 
 22  apps you soon realize that two types of issues can make debugging painful:  
 23   
 24  1. typos are too easy, yet will create new topics,  
 25          causing your messages to never reach one or more (even all) of your  
 26          listeners.  
 27  2. By default, pubsub infers from the first sendMessage or subscribe (whichever 
 28          is execute first during a run) what the message arguments are. Thereafter,  
 29          all other calls to sendMessage and subscribe must adhere to the same kwarg  
 30          names. A typo in the arg name will cause an en exception to get thrown,  
 31          although the fault may in fact be in the first sender or subscription for 
 32          the topic used.  
 33   
 34  To help with #2, it is possible to create a hierarchy of topic definitions, 
 35  which contain the topic names and a signature that all listeners of the 
 36  topic will have to satisfy. To help with #1, setTopicUnspecfiedFatal() can be  
 37  called after pubsub has been imported, then only topics that have a definition  
 38  are allowed; typos will cause an exception.  
 39   
 40  The format of this tree can be seen by calling 
 41  exportTopicTreeSpec(moduleName='yourModule'); then look at the contents of the 
 42  yourModule.py.  
 43   
 44  A typical workflow would be: 
 45   
 46  - use exportTopicTreeSpec() once in a while and inspect the generated 
 47    file for topics that differ only by a typo, fix issues 
 48  - once the topic tree has started to gel, save the topic tree file 
 49    created by exportTopicTreeSpec() and complete it with documentation 
 50  - use addTopicDefnProvider() and setTopicUnspecifiedFatal() 
 51  - to add new topics, manually add them to the topic tree file, OR 
 52    comment out the call to setTopicUnspecifiedFatal() and recall 
 53    exportTopicTreeSpec (your docs should *not* be affected but YMMV) 
 54     
 55  Note that by importing the topic tree file, some editors will provide  
 56  code completion on topic names. Example:: 
 57   
 58          from pubsub import pub 
 59          import your_topic_tree_module 
 60          pub.addTopicDefnProvider(your_topic_tree_module) 
 61          pub.setTopicUnspecifiedFatal() 
 62   
 63          pub.subscribe(listener, your_topic_tree_module.topic1) 
 64          pub.sendMessage(your_topic_tree_module.topic1.subtopic1, kwarg1=..., ...) 
 65   
 66    TODO: add isMsgReceivable(listener, topicName) to find out if listener is 
 67          subscribed to topicName or any of its subtopics.
